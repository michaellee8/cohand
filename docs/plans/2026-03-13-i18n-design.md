# i18n Design: react-i18next with EN / zh-TW / zh-CN

**Date:** 2026-03-13
**Status:** Approved
**Locales:** English (`en`), Traditional Chinese (`zh-TW`), Simplified Chinese (`zh-CN`)

---

## Architecture & Setup

**Dependencies:** `i18next` + `react-i18next`. No plugins (no lazy loading, no backend loader).

**Init file (`src/i18n/index.ts`):** Calls `i18next.use(initReactI18next).init()` with:
- `resources`: statically imported from 3 locale JSON files
- `lng`: read from `chrome.storage.local` (settings.language), falling back to `navigator.language`, then `'en'`
- `fallbackLng: 'en'`
- `interpolation: { escapeValue: false }` (React already escapes)

**Bootstrap:** `src/entrypoints/sidepanel/index.tsx` imports `../../i18n` before rendering `<App />`. Init is synchronous with English, then calls `i18next.changeLanguage()` once storage resolves. No loading spinner needed.

**Settings dropdown:** Replace the dead text input with a `<select>` offering `English`, `繁體中文`, `简体中文`. On change: call `i18next.changeLanguage(lng)` + persist via `updateSettings({ language: lng })`.

**Language switch:** Immediate re-render (no reload). react-i18next's `I18nextProvider` context handles this automatically.

## What Gets Translated vs. What Stays Untranslated

**Translated (static UI):**
- Tab labels, page headings, section titles
- Button text, placeholder text
- Wizard step titles and instructions
- Error banners (static messages)
- `aria-label` attributes (accessibility)

**Untranslated (dynamic content):**
- Task names (user-authored)
- LLM chat responses
- Script source code
- Runtime error messages from service worker
- Domain names, timestamps
- Console logs, internal error throws

## Locale Detection & Persistence

**First launch (no stored setting):**
1. Init synchronously with `lng: 'en'`
2. Read `chrome.storage.local` for `settings.language`
3. If no stored value, detect from `navigator.language`:
   - `zh-TW`, `zh-Hant` → `zh-TW`
   - `zh-CN`, `zh-Hans`, `zh` → `zh-CN`
   - Everything else → `en`
4. Call `i18next.changeLanguage(detected)` and persist to storage

**Subsequent launches:** Read stored value, call `changeLanguage()`.

**Mapping function:**
```ts
function resolveLocale(raw: string): 'en' | 'zh-TW' | 'zh-CN' {
  const lower = raw.toLowerCase();
  if (lower.startsWith('zh-tw') || lower.startsWith('zh-hant')) return 'zh-TW';
  if (lower.startsWith('zh')) return 'zh-CN';
  return 'en';
}
```

**Legacy migration:** Old freeform string values (e.g., `'ja'`) map to `'en'` via the fallback.

The `settings.language` type changes from `string` to `'en' | 'zh-TW' | 'zh-CN'`.

## Testing

**Unit tests (`src/i18n/index.test.ts`):**
- `resolveLocale()` mapping correctness
- i18next init loads all 3 locales without errors
- `changeLanguage()` switches and `t()` returns correct string

**Translation completeness check:**
- Script or test that loads all 3 JSON files, compares key sets, fails if any key missing from zh-TW or zh-CN

**E2E tests:**
- Update existing Settings test to use dropdown with valid values
- New test: change to `zh-TW`, verify a known string renders in Chinese

**Not tested:** Translation quality (human review), every string in every locale (completeness check covers key presence).
